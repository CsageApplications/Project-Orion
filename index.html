Vision

Why we're building this

Orion lowers the barrier to home robotics. Instead of buying one expensive all-in-one robot, customers start with an affordable intelligent head and expand only when they need new capability.

Observation

Robots are all-or-nothing

Most home robot concepts require a full hardware commitment before you know if the system actually fits your life.

Exploration

What if it was modular?

Start with an intelligent head, then attach mobility, cleaning, arms, or tools only when you actually need them.

Approach

Software-first, hardware-ready

Build the brain, personality, and coordination layer first. Design hardware contracts so modules can plug in later.

Platform

Orion Head

Base product

Orion Head Developer Kit

Affordable entry point

The head should be the smallest possible commitment — a useful standalone device even before any modules are added.

Core promise

One brain, many bodies

The same Orion Head becomes a desk assistant, patrol robot, vacuum robot, yard helper, or specialty platform.

Architecture

Platform architecture

Orion is not one robot. It is a modular robotics operating system with a standard way to discover, validate, and control capabilities.

React Command Center Home screen for chat, live camera, telemetry, command buttons, logs, routines, and module management.
Rust Backend / Orion Core Axum APIs, WebSockets, command queue, auth, memory, safety gates, telemetry, and LLM orchestration.
Orion OS Runtime Containerized runtime on the Orion Head: speech, vision, task planning, module discovery, and OTA updates.
ROS 2 Robot Layer Nodes for camera, audio, motion, navigation, behavior, battery, sensors, and telemetry.
Microcontroller Layer ESP32/STM32 for real-time motor control, servos, encoders, LEDs, sensors, and emergency stop.
Accessory Modules Each module exposes a capability contract: vacuum, sweep, carry, patrol, shovel, dock, follow, or manipulate.
Contracts

Capability contract

Every module tells Orion what it is, what it can do, what risks exist, and what commands it accepts.

Example: Vacuum Module

{ "module": "vacuum", "version": "1.0", "capabilities": [ "vacuum_room", "spot_clean", "return_to_dock", "report_bin_status" ], "risk_level": "medium", "requires_confirmation": false }

Why this matters

This is what makes the platform extensible. If Orion understands modules through standard contracts, any new hardware can describe its own capabilities rather than requiring custom code for each one.

Roadmap

Build phases

Start with a magical demo, evolve into a developer platform, and eventually build a consumer ecosystem.

Conversational Orion Head

Voice in, LLM reasoning, voice out, expressive face, React dashboard, and memory.

Embodied expression

Animated eyes, head pan/tilt, camera awareness, wake word, and personality system.

Module interface

Design electrical/mechanical expansion standard and software capability contracts.

Mobility + cleaning

Add mobile base, docking, patrol mode, vacuum/sweeper modules, and safety boundaries.

Developer kit

Package Orion Head + SDK + docs + sample accessories for early builders and testers.

Marketplace ecosystem

Third-party modules, skills, routines, subscription services, and specialized verticals.
